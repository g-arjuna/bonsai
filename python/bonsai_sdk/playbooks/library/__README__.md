# Playbook Library

YAML playbook definitions consumed by `catalog.py` and `executor.py`.

## Authoring workflow

Each file covers one `detection_rule_id`. See `bgp_session_down.yaml` for the
canonical schema example.

Files in this directory are either:
- **Hand-authored** — written by an engineer who validated the gNMI path in the lab
- **Harvest-produced** — generated by the Codex harvest workflow in `PHASE5_HARVEST_PROMPT.md`,
  then human-reviewed before committing

Mark every entry with `verified_on` metadata once validated in ContainerLab.
Entries with `verified_on: []` are drafts — do not enable auto-remediation for them.

## YAML schema

```yaml
detection_rule_id: <string>    # matches Detection.rule_id in the rule engine
playbooks:
  - name: <string>
    vendor: <string>            # nokia_srl | cisco_xrd | arista_ceos | juniper_crpd | "*" (any)
    description: <string>
    risk_tier: safe | disruptive | last_resort
    preconditions:              # Python expressions evaluated against detection.features
      - "<expr>"
    steps:
      - gnmi_set:
          path: "<YANG path with {feature_field} substitution>"
          value: "<RFC 7951 JSON string>"
      - sleep: <seconds>
    verification:
      wait_seconds: <int>
      cypher: "<Cypher query returning truthy value when recovery confirmed>"
    verified_on:
      - firmware: "<version string>"
        last_tested: "<YYYY-MM-DD>"
        tested_by: "<name>"
    confidence: high | medium | low | unverified
```

## Adding a new playbook

1. Write the YAML (use `bgp_session_down.yaml` as the template)
2. Validate the gNMI path manually via `gnmic` against the target device
3. Add `verified_on` metadata
4. Write an ADR entry in `DECISIONS.md` if it introduces a new operation category
