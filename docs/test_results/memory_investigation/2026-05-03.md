# Memory Growth Investigation ‚Äî 2026-05-03

**Bonsai version**: a46544d ‚Üí 073fe8b (v11 Sprint 1 base)
**Observation**: RSS grew to ~9 GB over a multi-hour ContainerLab session (4 devices, telemetry flowing).
**WAL size at observation**: 15 MB (`bonsai.db.wal`)
**bonsai startup time after observation**: 289 seconds (WAL replay)

---

## Hypothesis ranking

The following components were examined in order of most-likely-to-least-likely contribution.

### H1: LadybugDB (lbug) graph state accumulation ‚Äî HIGH CONFIDENCE ‚öÝÔ∏è

**Evidence**:
- `bonsai.db.wal` grew to 15 MB during a multi-hour session.
- Startup WAL replay takes 289 seconds ‚Äî this is proportional to accumulated state.
- The retention module (`src/retention.rs`) defaults to `enabled = false`. With retention disabled, `StateChangeEvent` nodes accumulate without bound in the lbug graph.
- Each telemetry update that changes observed state writes a `StateChangeEvent` node. At 4 devices √ó ~5 gNMI paths √ó 5-second poll cadence, that's ~240 events/minute, or ~14,400/hour.
- After 10 hours: ~144,000 StateChangeEvents in the graph, each with edges and metadata in lbug's in-memory read cache.
- lbug's cache has no configured cap (defaults to an internal heuristic, likely proportional to DB size).

**Fix**: Enable retention by default (`enabled = true`). Lower defaults to `max_age_hours = 24`, `max_state_change_events = 10000`. This bounds the graph to a rolling 24-hour window.

**Status**: Fixed in v11 Sprint 2 ‚Äî see `bonsai.toml.example` updated defaults.

---

### H2: Broadcast channel slow subscriber ‚Äî MEDIUM CONFIDENCE

**Evidence**:
- The event bus uses a `tokio::sync::broadcast` channel with capacity 2048.
- With 8+ subscribers (graph writer, archive, detector, retention, SSE handler(s), output adapters), a single slow subscriber causes the channel to fill.
- A full channel causes `send()` to drop the oldest message, but the messages already in flight stay in memory until all subscribers have consumed them.
- During the observation session, at least 3 output adapters were registered (prometheus, splunk, elastic) plus the SSE handler in the UI. If the SSE handler (or an adapter) stalled, the channel would hold 2048 √ó `size_of::<TelemetryUpdate>()` bytes √ó number of lagging receivers.
- `size_of::<TelemetryUpdate>()` is ~120 bytes struct + heap for `String` fields and `serde_json::Value`. Realistic: 400‚Äì2000 bytes per update depending on value complexity.
- Worst case at full capacity with 3 lagging receivers: 2048 √ó 3 √ó 2000 = ~12 MB. Not 9 GB. Not the primary culprit.

**Fix**: Added slow-subscriber warning (`SLOW_SUBSCRIBER_WARN_THRESHOLD_PCT = 50%`) to surface lagging subscribers before they cause pressure. Metric: `bonsai_event_bus_slow_subscriber_warnings_total`.

**Status**: Monitoring added in v11 Sprint 2. Will confirm via memory profiler runs.

---

### H3: Archive partition writer accumulation ‚Äî LOW CONFIDENCE

**Evidence**:
- `HourlyArchiveWriter` keeps one `OpenPartitionWriter` per `(year, month, day, hour, target)` combination.
- Partition writers are closed when a newer hour's data arrives (`close_hours_before`).
- With 4 devices √ó 1 open hour = 4 writers at any time. Each `ArrowWriter` buffers up to `max_batch_rows = 1000` rows before flushing.
- Maximum memory from this: 4 √ó 1000 √ó 2000 bytes = ~8 MB. Not the culprit.
- **Exception**: if telemetry timestamps are irregular (device clock drift, replay), multiple hour partitions can be open simultaneously and never roll over. This would hold writers indefinitely.

**Fix**: Added `writer_max_idle_secs = 7200` ‚Äî partition writers idle for >2h are force-closed on a background timer.

**Status**: Fixed in v11 Sprint 2.

---

### H4: Counter summarizer state accumulation ‚Äî LOW CONFIDENCE

**Evidence**:
- `CounterSummarizer` keeps one `InterfaceWindow` per `(target, interface_name)`.
- With 4 devices √ó ~5 interfaces = ~20 entries max. Each window holds ~10 `CounterStats` structs.
- Maximum: 20 √ó 10 √ó 80 bytes = ~16 KB. Negligible.
- `flush_stale()` is called on a timer, clearing idle windows. No unbounded growth path.

**Fix**: Added `max_entries = 1024` cap with oldest-idle eviction as a defensive bound. This prevents pathological growth in hypothetical future scenarios (many devices, dynamically discovered interfaces).

**Status**: Defensive bound added in v11 Sprint 2.

---

## Root cause conclusion

**Primary culprit**: LadybugDB accumulating `StateChangeEvent` nodes without retention bounds. With `retention.enabled = false` (old default), the graph grows linearly with session duration. At 10 hours with 4 devices, the graph accumulates ~144,000 StateChangeEvent nodes plus their edges and property maps in lbug's cache.

**Secondary culprit**: Not yet confirmed, but lbug's read cache size is not explicitly configured and may grow proportionally to graph size. Profiling with the new `bonsai_memory_rss_bytes` metric and memory snapshots should confirm after the retention fix.

## Fixes applied (v11 Sprint 2)

| Component | Fix | Config |
|-----------|-----|--------|
| Retention | Enable by default; lower defaults | `[retention] enabled=true, max_age_hours=24, max_state_change_events=10000` |
| Archive writers | Max-idle close on 2h timer | `[archive] writer_max_idle_secs=7200` |
| Counter summarizer | Hard cap with oldest-idle eviction | `max_entries=1024` |
| Event bus | Slow-subscriber warning at 50% fill | Metric: `bonsai_event_bus_slow_subscriber_warnings_total` |
| Memory profiler | Background sampler emits `bonsai_memory_rss_bytes` | Always on for core role |
| Disk guard | Bounds archive + graph dir sizes | `[storage] max_archive_bytes, max_graph_bytes` |
| Archive compression | ZSTD level 12 + dictionary encoding | `[archive] compression_level=12` |
